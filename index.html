Repair Service Management

- Web Application -
The purpose of this application is to be used by the employees of a company that repairs different types of electronic devices. The application can be used by several types of users, each of them having different roles and permissions.

App description:
Client comes to a service with a broken device.
A front-desk employee logs in the application, registers client and device details and opens a Registry Service file where he assigns the device to a specific warehouse.

Another user log's in (as a technician), where he can view all the Service Registry's.
He can create multiple Repair Registries for a a Service Registry where he can describe the details about the repairs: establishing the defect, estimated time for repair, replaced parts(individual object with name, price), workmanship(individual object with amount of hours, price, operations), a total cost (automatically calculated based on the workmanship and parts price) and change the status(open, pending, close).
The Repair Registry status reflects over the front-desk user and when a Service Registry is closed the owner is contacted. User Manager can view reports and statistics about the type of devices that comes to the service, about internal costs and profit. User Admin can create new users, reset passwords and have access to all features of the application.

Technologies:
BACK-END:

Language: C#
Frameworks: ASP.NET Web-API, Entity Framework, NUnit
Database: MS SQL Server

FRONT-END:

Frameworks: ANGULAR 7, BOOTSTRAP
Presentation Languages: HTML & CSS

Train routes

- Desktop Application -
I have developed an application that manages the routes of trains.
Within the app the user can add a passenger route and input the details for that route: start city, destination city, departure time, arrival time, number of wagons. The user can also add a cargo route and here he can also add weight for the train. The user can also delete a route.
The program always displays a list of routes with all the trains, that is automatically refreshed after each add/delete of a route. The user can sort the routes, export them in a file and determine if there is a round - trip between 2 cities.
Interaction with the user was made through a GUI developed in SWING.
